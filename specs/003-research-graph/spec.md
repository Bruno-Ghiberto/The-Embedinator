# Feature Specification: ResearchGraph

**Feature Branch**: `003-research-graph`
**Created**: 2026-03-11
**Status**: Draft
**Input**: User description: "ResearchGraph - Layer 2 of the three-layer agent architecture. Worker layer that takes a single sub-question, searches document collections using hybrid retrieval, reranks results, tracks deduplication, and iterates until confidence threshold or search budget exhaustion."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Evidence Retrieval for a Sub-Question (Priority: P1)

A user asks a question that the conversation layer decomposes into sub-questions. For each sub-question, the system spawns a research worker that searches the user's document collections, finds relevant evidence, and returns a packaged answer with citations and a confidence score.

**Why this priority**: This is the core value proposition — without evidence retrieval, the system cannot answer questions grounded in the user's documents. Every other capability depends on this fundamental search-and-answer loop.

**Independent Test**: Can be fully tested by sending a sub-question with a target collection containing known documents and verifying the system returns relevant chunks, a coherent answer, citations pointing to source documents, and a confidence score reflecting retrieval quality.

**Acceptance Scenarios**:

1. **Given** a sub-question and a collection with relevant documents, **When** the research worker executes, **Then** it returns an answer with at least one citation and a confidence score above the threshold.
2. **Given** a sub-question and a collection with relevant documents, **When** the research worker finds sufficient evidence on the first search, **Then** it terminates early without exhausting the full search budget.
3. **Given** a sub-question, **When** the research worker completes, **Then** the answer includes citations that reference specific source documents and chunk locations.

---

### User Story 2 - Iterative Search Refinement (Priority: P2)

When initial search results are insufficient, the system iteratively refines its search strategy — rephrasing queries, narrowing by metadata filters, searching across multiple collections, and retrieving parent context for promising child chunks — until it accumulates enough evidence or exhausts its search budget.

**Why this priority**: Many real-world questions cannot be answered by a single search. The ability to refine and try multiple search angles is what distinguishes a useful research system from a basic keyword search.

**Independent Test**: Can be tested by sending a sub-question where the answer requires combining evidence from multiple searches (e.g., initial search returns partial results, requiring a follow-up with different filters or query rephrasing).

**Acceptance Scenarios**:

1. **Given** a sub-question where the first search returns low-relevance results, **When** the research worker iterates, **Then** it tries alternative search strategies (query rephrasing, metadata filtering, cross-collection search) and accumulates better evidence.
2. **Given** a sub-question targeting multiple collections, **When** the research worker searches, **Then** it can fan out across all enabled collections and merge results.
3. **Given** a sub-question, **When** child chunks are found, **Then** the system can retrieve their parent chunks for broader context.

---

### User Story 3 - Search Budget Enforcement (Priority: P2)

The system enforces strict limits on the number of search iterations and tool calls to prevent runaway resource consumption. It stops when it reaches either the iteration limit or the tool call limit, regardless of confidence level.

**Why this priority**: Without budget limits, a difficult question could cause unbounded resource consumption (search calls, compute time, token usage). Budget enforcement is essential for predictable system behavior and cost control.

**Independent Test**: Can be tested by sending a sub-question with no matching documents (forcing the system to exhaust its budget) and verifying it terminates within the defined limits.

**Acceptance Scenarios**:

1. **Given** a sub-question, **When** the research worker reaches the maximum iteration count (10), **Then** it stops iterating and proceeds to answer packaging.
2. **Given** a sub-question, **When** the research worker reaches the maximum tool call count (8), **Then** it stops calling tools and proceeds to answer packaging.
3. **Given** a sub-question, **When** no new tool calls are generated by the orchestrator, **Then** the research worker terminates the loop (tool exhaustion).

---

### User Story 4 - Deduplication of Search Results (Priority: P3)

The system tracks which (query, document chunk) pairs have already been retrieved and skips duplicates across iterations. This prevents the same evidence from being retrieved repeatedly, wasting budget and polluting results.

**Why this priority**: Without deduplication, iterative searches would re-retrieve the same chunks, wasting tool calls and inflating result sets with redundant content. Deduplication ensures each iteration adds genuinely new evidence.

**Independent Test**: Can be tested by running a research worker where the same query returns overlapping results across iterations, and verifying duplicate chunks are not added to the result set.

**Acceptance Scenarios**:

1. **Given** a sub-question where two iterations return overlapping chunks, **When** results are merged, **Then** each unique (query, parent chunk) pair appears only once.
2. **Given** a previously seen (query, parent chunk) pair, **When** it is encountered again, **Then** the system skips it without counting against the tool call budget.

---

### User Story 5 - Graceful Degradation on Low Confidence (Priority: P3)

When the research worker exhausts its search budget but has not reached sufficient confidence, it either routes to a recovery mechanism (meta-reasoning) or generates a transparent "insufficient information" response rather than fabricating an answer.

**Why this priority**: Users must be able to trust the system's answers. When evidence is insufficient, it is far better to say "I don't have enough information" than to hallucinate. This maintains user trust and system integrity.

**Independent Test**: Can be tested by sending a sub-question with no matching documents in any collection and verifying the system returns a graceful fallback response with low confidence, not a fabricated answer.

**Acceptance Scenarios**:

1. **Given** a sub-question with no relevant documents available, **When** the search budget is exhausted, **Then** the system returns a fallback response indicating insufficient information.
2. **Given** a sub-question where confidence remains below the threshold after budget exhaustion, **When** the meta-reasoning recovery system is available, **Then** the research worker routes to it for failure recovery.
3. **Given** a sub-question where confidence remains below threshold after budget exhaustion, **When** the meta-reasoning recovery system is NOT available, **Then** the research worker generates a graceful fallback response.

---

### User Story 6 - Context Compression Under Load (Priority: P3)

When the accumulated evidence from multiple search iterations grows large enough to approach processing limits, the system compresses (summarizes) the retrieved context before continuing. This allows the research worker to continue searching without hitting capacity limits.

**Why this priority**: Long research sessions can accumulate large volumes of retrieved text. Without compression, the system would hit processing limits and fail mid-research. Compression allows the research to continue while preserving the essential information.

**Independent Test**: Can be tested by running a research worker that accumulates a large volume of chunks across many iterations and verifying that compression triggers before capacity is exceeded, and the research continues successfully.

**Acceptance Scenarios**:

1. **Given** a research session that has accumulated many chunks, **When** the total token count reaches 75% of the model's context window, **Then** the system compresses the retrieved context into a summary.
2. **Given** compressed context, **When** the research worker continues searching, **Then** it operates on the compressed context and can still accumulate new results.
3. **Given** a compression event, **When** the answer is finally packaged, **Then** the answer quality reflects the full body of evidence (not just post-compression results).

---

### Edge Cases

- What happens when the target collection does not exist or is empty? The system should return an empty result set with zero confidence, not an error.
- What happens when all document collections are disabled? The system should return a fallback response immediately without attempting searches.
- What happens when the search service is temporarily unavailable? The system retries the failed call once. Both the original attempt and the retry count against the tool call budget. If the retry also fails, the system moves to the next iteration or fallback.
- What happens when context compression produces a summary that loses critical details? The system should preserve citation references even after compression.
- What happens when the orchestrator generates an invalid or unsupported tool call? The system should skip the invalid call, log it, and continue with the next iteration.
- What happens when multiple research workers are spawned concurrently for different sub-questions? Each worker must operate independently with its own state and deduplication tracking.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST accept a single sub-question and target collection(s) as input and produce an answer with citations and a confidence score as output.
- **FR-002**: System MUST use an orchestrator to decide which search tools to call next based on the current sub-question, retrieved evidence, and remaining budget.
- **FR-003**: System MUST support six search tools: child chunk search, parent chunk retrieval, cross-encoder reranking, collection filtering, metadata filtering, and cross-collection search.
- **FR-004**: System MUST enforce a maximum of 10 loop iterations per research session.
- **FR-005**: System MUST enforce a maximum of 8 tool calls per research session.
- **FR-006**: System MUST terminate the search loop when the orchestrator determines it has sufficient evidence (confidence above threshold).
- **FR-007**: System MUST terminate the search loop when the orchestrator generates no new tool calls (tool exhaustion).
- **FR-008**: System MUST maintain a deduplication index of (normalized query, parent chunk ID) pairs and skip any pair already seen in the current session.
- **FR-009**: System MUST compute confidence scores from measurable retrieval signals (rerank scores, chunk count, coverage), not from self-assessment.
- **FR-010**: System MUST compress retrieved context when the accumulated token count reaches 75% of the model's context window capacity.
- **FR-011**: System MUST preserve citation references through context compression events.
- **FR-012**: System MUST route to the meta-reasoning recovery system when confidence is below threshold and the search budget is exhausted.
- **FR-013**: System MUST generate a graceful "insufficient information" fallback response when tools are exhausted and meta-reasoning is not available.
- **FR-014**: System MUST package the final answer with answer text, citations referencing source documents, the list of retrieved evidence, and a confidence score (integer 0-100).
- **FR-015**: System MUST support concurrent execution — multiple research workers for different sub-questions operate independently with isolated state.
- **FR-016**: System MUST retry a failed tool call exactly once before moving on. Both the original attempt and the retry count against the tool call budget.
- **FR-017**: System MUST emit structured log events at key decision points: loop start/end, each tool call (with result status), context compression events, confidence score changes, and fallback triggers.

### Key Entities

- **Sub-Question**: A focused research query decomposed from the user's original question. Attributes: question text, target collections, session context.
- **Retrieved Chunk**: A piece of evidence found during search. Attributes: content text, source document reference, relevance score, parent chunk reference, collection of origin.
- **Citation**: A reference to the specific source of a piece of evidence. Attributes: source document, chunk location, relevance to the answer.
- **Research Answer**: The packaged output of a research session. Attributes: answer text, list of citations, list of retrieved chunks, confidence score (0-100).
- **Deduplication Key**: A unique identifier for a (query, parent chunk) pair used to prevent redundant retrievals.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Research workers return answers with at least one citation for 95% of sub-questions where relevant documents exist in the collection.
- **SC-002**: No research session exceeds 10 iterations or 8 tool calls, enforced unconditionally.
- **SC-003**: Duplicate (query, chunk) pairs are never included in the final result set — deduplication is 100% effective.
- **SC-004**: Context compression triggers at 75% of model context window capacity, preventing mid-research failures in 100% of cases.
- **SC-005**: When relevant documents exist, confidence scores computed from retrieval signals correlate with actual answer quality (higher-confidence answers are more accurate).
- **SC-006**: When no relevant documents exist, the system returns a fallback response (not a fabricated answer) within the budget limits.
- **SC-007**: Multiple concurrent research workers for different sub-questions produce independent, correct results without cross-contamination.
- **SC-008**: End-to-end research session (from sub-question to packaged answer) completes within 30 seconds for typical queries against collections with fewer than 100,000 chunks.

## Clarifications

### Session 2026-03-11

- Q: When a search service call fails, should the system retry or fail fast? How do failures count against the tool call budget? → A: Retry once; count both attempts (original + retry) against the 8-call budget.
- Q: Should research sessions emit observable signals for monitoring and debugging? → A: Structured logging at key decision points (loop lifecycle, tool calls, compression, fallback triggers).
- Q: At what capacity percentage should context compression trigger? → A: 75% of model context window, leaving 25% headroom for continued search.

## Assumptions

- The conversation layer (Layer 1) correctly decomposes user questions into sub-questions and provides valid collection names.
- Document collections have already been ingested and indexed with both dense and sparse vectors for hybrid search.
- A cross-encoder model is available for reranking results.
- The meta-reasoning recovery system (Layer 3) may not be available in the initial release; the fallback response path must work independently.
- The confidence threshold (0.6 on a 0-1 scale) is a sensible default but may be tuned based on real-world usage.
- Token counting for compression decisions uses an approximate method that is accurate enough for threshold-based decisions.

## Dependencies

- **Layer 1 (ConversationGraph)**: Spawns research workers and consumes their packaged answers.
- **Layer 3 (MetaReasoningGraph)**: Receives low-confidence results for recovery. May not be available in initial release.
- **Document Ingestion**: Collections must be populated before research can find results.
- **Search Infrastructure**: Hybrid search (dense + sparse) and cross-encoder reranking must be operational.
