## 1. Architecture Overview

The TUI installer is a single statically-linked Go binary named `embedinator` that replaces the
`embedinator.sh` and `embedinator.ps1` scripts as the primary entry point for The Embedinator. The
user installs the binary via a one-liner, then runs it to configure, install, and manage the full
Docker Compose stack.

```
User's machine
+-----------------------------------------------------------------------+
|                                                                       |
|  embedinator (Go binary)                                              |
|  +-------------------------------------------------------------------+
|  |                                                                   |
|  |  CLI Layer (cobra)                                                |
|  |  Parses subcommands: setup, start, stop, status, logs, config,   |
|  |  update, doctor                                                   |
|  |  +---------------------------------------------------------------+
|  |  |                                                               |
|  |  |  TUI Layer (bubbletea + lipgloss + bubbles)                   |
|  |  |  Multi-step wizard with progress, spinners, selection         |
|  |  |  Only activated for interactive commands (setup, config)       |
|  |  |                                                               |
|  |  +---------------------------------------------------------------+
|  |  |                                                               |
|  |  |  Engine Layer (internal packages)                              |
|  |  |  +------------------+  +------------------+  +---------------+|
|  |  |  | preflight        |  | docker           |  | config        ||
|  |  |  | - Docker check   |  | - compose up/dn  |  | - read/write  ||
|  |  |  | - disk space     |  | - health poll    |  | - .env gen    ||
|  |  |  | - RAM check      |  | - image pull     |  | - config.yaml ||
|  |  |  | - port scan      |  | - logs stream    |  | - Fernet key  ||
|  |  |  +------------------+  +------------------+  +---------------+|
|  |  |  +------------------+  +------------------+  +---------------+|
|  |  |  | gpu              |  | ollama           |  | sysinfo       ||
|  |  |  | - detect NVIDIA  |  | - detect local   |  | - OS detect   ||
|  |  |  | - detect AMD     |  | - model pull     |  | - arch detect ||
|  |  |  | - detect Intel   |  | - health check   |  | - WSL2 detect ||
|  |  |  | - macOS Metal    |  | - version check  |  | - env info    ||
|  |  |  +------------------+  +------------------+  +---------------+|
|  |  |                                                               |
|  |  +---------------------------------------------------------------+
|  |                                                                   |
|  +-------------------------------------------------------------------+
|                          |                                            |
|                          v                                            |
|  +-------------------------------------------------------------------+
|  |  Docker Compose Stack                                             |
|  |  +----------+ +----------+ +-----------+ +-----------+            |
|  |  | qdrant   | | ollama   | | backend   | | frontend  |            |
|  |  | :6333    | | :11434   | | :8000     | | :3000     |            |
|  |  +----------+ +----------+ +-----------+ +-----------+            |
|  +-------------------------------------------------------------------+
|  |  Config Files (generated by TUI)                                  |

|  |  +----------+ +------------------+ +-----------------------------+|
|  +-------------------------------------------------------------------+

+-----------------------------------------------------------------------+
```

**Data flow for initial setup:**

```
User runs `embedinator` (no args)
  |
  v
Is config.yaml present AND valid?
  |
  +-- NO --> Launch TUI wizard (full setup)
  |            |
  |            v
  |          [Welcome] -> [Prerequisites] -> [Ollama] -> [Ports]
  |            -> [GPU] -> [Models] -> [API Keys] -> [Summary] -> [Install]
  |            -> [Complete]
  |            |
  |            v
  |          Writes: .env, config.yaml, optional overlays
  |          Runs: docker compose up --build -d
  |          Polls health, pulls models
  |
  +-- YES -> Is stack running?
               |
               +-- YES -> Print status and URLs
               +-- NO  -> Run `embedinator start` (non-interactive)
```

## 2. Technology Justification

### Why Go

| Factor | Go | Rust | Python | Node.js |
|--------|-----|------|--------|---------|
| Single binary, zero deps | Yes (static linking, CGO_ENABLED=0) | Yes | No (needs Python runtime) | No (needs Node runtime) |
| Cross-compile for 6+ targets | `GOOS/GOARCH` trivial | Possible but harder (cross-linker) | N/A | N/A |
| TUI ecosystem maturity | Charm stack (bubbletea, lipgloss, bubbles) is production-grade, actively maintained, and beautiful | crossterm/ratatui is solid but more boilerplate | Rich/Textual are good but require Python | ink/blessed are fragile |
| Docker SDK | `docker/docker` client library, native | bollard crate, less mature | docker-py, good but needs runtime | dockerode, needs runtime |
| Build speed | Fast (2-5s incremental) | Slow (30s+) | N/A | N/A |
| goreleaser integration | First-class (Go-native) | Possible but not native | N/A | N/A |
| Contributor accessibility | Moderate learning curve | Steep learning curve | Easy but wrong tool | Easy but wrong tool |

**Decision: Go.**

The Charm ecosystem (charmbracelet/bubbletea, charmbracelet/lipgloss, charmbracelet/bubbles) is the
decisive factor. It produces terminal UIs with the visual polish of a native application: smooth
animations, ANSI color gradients, styled borders, progress bars, spinners, and multi-select lists --
all composable via the Elm Architecture (Model-Update-View). No other ecosystem achieves this with
zero runtime dependencies.

### Charm Stack Components

| Package | Purpose | Version |
|---------|---------|---------|
| `charmbracelet/bubbletea` | TUI framework (Elm Architecture) | v2 |
| `charmbracelet/lipgloss` | Terminal styling (colors, borders, padding) | v2 |
| `charmbracelet/bubbles` | Pre-built components (spinner, progress, text input, list, table) | v0.20+ |
| `charmbracelet/huh` | Form-based wizard screens (select, confirm, input, multi-select) | v0.6+ |
| `spf13/cobra` | CLI subcommand framework | v1.8+ |
| `docker/docker` | Docker Engine API client | v27+ |

### Why NOT a TUI framework on top of the existing Python backend

The TUI binary runs BEFORE Docker is started. It cannot depend on the Python backend, Node.js
frontend, or any containerized service. It must be a standalone executable that works with nothing
but Docker installed. This rules out any language that requires a runtime.

## 3. Distribution Methods

### 3.1 Method 1: curl one-liner (Linux / macOS)

```bash
curl -fsSL https://embedinator.dev/install.sh | bash
```

The install script does the following:

```
install.sh:
  1. Detect OS (linux, darwin) and ARCH (amd64, arm64)
  2. Determine latest version from GitHub Releases API:
     GET https://api.github.com/repos/Bruno-Ghiberto/The-Embedinator/releases/latest
  3. Download binary:
     https://github.com/Bruno-Ghiberto/The-Embedinator/releases/download/v{VERSION}/embedinator_{OS}_{ARCH}.tar.gz
  4. Verify checksum:
     Download checksums.txt, verify SHA-256
  5. Extract binary to /usr/local/bin/embedinator (or ~/.local/bin/embedinator if no sudo)
  6. chmod +x
  7. Verify: embedinator --version
  8. Print success message with next steps
```

**Fallback path for no-sudo environments:**

```
If /usr/local/bin is not writable:
  Install to ~/.local/bin/embedinator
  Check if ~/.local/bin is in PATH
  If not: print "Add to PATH" instructions for bash/zsh/fish
```

**Script safety considerations:**

- The script is hosted at a stable URL (GitHub Pages or raw.githubusercontent.com)
- Users can download and inspect before running: `curl -fsSL url > install.sh && less install.sh && bash install.sh`
- The script sets `set -euo pipefail`
- Each step is logged with clear messages
- Checksum verification is mandatory -- abort on mismatch

### 3.2 Method 2: PowerShell one-liner (Windows)

```powershell
irm https://embedinator.dev/install.ps1 | iex
```

The PowerShell install script:

```
install.ps1:
  1. Detect ARCH (amd64, arm64)
  2. Query GitHub Releases API for latest version
  3. Download embedinator_windows_amd64.zip
  4. Verify SHA-256 checksum
  5. Extract embedinator.exe to:
     a. $env:LOCALAPPDATA\Programs\embedinator\embedinator.exe
     b. Add that directory to user PATH (via [Environment]::SetEnvironmentVariable)
  6. Verify: embedinator --version
  7. Print success message
```

**PATH handling on Windows:**

The binary goes into `%LOCALAPPDATA%\Programs\embedinator\` rather than a system directory. The
script appends this to the user's PATH via the registry (persistent across sessions). A new
terminal window is required for the PATH change to take effect -- the script prints this warning.

### 3.3 Method 3: Homebrew (macOS / Linux)

```bash
brew install Bruno-Ghiberto/tap/embedinator
```

This requires a separate Homebrew tap repository: `github.com/Bruno-Ghiberto/homebrew-tap`.

The tap contains a formula auto-generated by goreleaser on each release:

```ruby
# Formula/embedinator.rb (auto-generated by goreleaser)
class Embedinator < Formula
  desc "TUI installer and manager for The Embedinator RAG system"
  homepage "https://github.com/Bruno-Ghiberto/The-Embedinator"
  version "0.3.0"

  on_macos do
    on_arm do
      url "https://github.com/.../embedinator_darwin_arm64.tar.gz"
      sha256 "..."
    end
    on_intel do
      url "https://github.com/.../embedinator_darwin_amd64.tar.gz"
      sha256 "..."
    end
  end

  on_linux do
    on_arm do
      url "https://github.com/.../embedinator_linux_arm64.tar.gz"
      sha256 "..."
    end
    on_intel do
      url "https://github.com/.../embedinator_linux_amd64.tar.gz"
      sha256 "..."
    end
  end

  def install
    bin.install "embedinator"
  end

  test do
    assert_match "embedinator version", shell_output("#{bin}/embedinator --version")
  end
end
```

### 3.4 Method 4: go install (Go developers)

```bash
go install github.com/Bruno-Ghiberto/The-Embedinator/cli@latest
```

This installs to `$GOPATH/bin` (typically `~/go/bin`). Only useful for users who already have Go
installed. Not the primary distribution path.

**Note:** The binary produced by `go install` is named `cli` (the Go module directory name). To get
the `embedinator` name, we use a `main.go` in `cli/cmd/embedinator/` as the entry point, so `go
install` targets `github.com/Bruno-Ghiberto/The-Embedinator/cli/cmd/embedinator@latest`.

### 3.5 Method 5: GitHub Releases Direct Download

Manual download from the Releases page. Each release includes:

```

embedinator_v0.3.0_linux_arm64.tar.gz
embedinator_v0.3.0_darwin_amd64.tar.gz
embedinator_v0.3.0_darwin_arm64.tar.gz
embedinator_v0.3.0_windows_amd64.zip
embedinator_v0.3.0_windows_arm64.zip
checksums.txt
```

## 4. TUI Screen-by-Screen Design

The wizard uses `charmbracelet/huh` forms for structured input screens and raw `bubbletea` models
for progress/animation screens. Each screen is a discrete `bubbletea.Model` wired into a parent
wizard model that manages navigation.

### 4.3 Screen 3: Ollama Configuration

```
  Ollama Configuration

  Ollama provides local AI inference. The Embedinator can use its own
  Ollama instance inside Docker, or connect to one you already have
  running locally.

  > Use Docker Ollama (recommended)
    Use my local Ollama (detected on port 11434, v0.6.2, Metal GPU)

    + Models stored in a Docker volume (separate from your local Ollama)

    + Uses your existing models

```

**Variant: macOS with local Ollama detected:**

```
  Ollama Configuration

  Local Ollama detected (v0.6.2, Apple Silicon Metal GPU)

  On macOS, local Ollama is STRONGLY recommended because Docker Desktop
  cannot pass through the Metal GPU. Local Ollama will be 3-10x faster
  for AI inference.

    Use my local Ollama (recommended for macOS)
  > Use Docker Ollama (CPU only, slower)
    Use a remote Ollama (enter URL manually)
```

**Variant: no local Ollama detected:**

```
  Ollama Configuration

  No local Ollama installation detected.

  > Use Docker Ollama (recommended)
    Use a remote Ollama (enter URL manually)
```

**Implementation notes:**

- Uses `huh.NewSelect()` for the three-option choice
- Local Ollama detection runs the same logic as `docs/design-launcher-improvements.md` section 3.2:
  check process, probe API, extract version and model list
- The "remote Ollama" option presents a `huh.NewInput()` for the URL with validation
- Selected choice is stored in the wizard state as `OllamaMode` (docker | local | remote)
- If `local` is chosen, `docker-compose.local-ollama.yml` will be generated later

- Multiple conflicts are resolved sequentially, one prompt per conflict
- Resolved ports are stored in wizard state

**No GPU detected:**

```

  GPU profile: none (CPU mode)

**Implementation notes:**

- NVIDIA detection: exec `nvidia-smi --query-gpu=name,driver_version --format=csv,noheader,nounits`
- AMD detection: check `/dev/kfd` existence, try `rocminfo`
- Intel detection: check `/dev/dri/renderD*` glob
- Docker NVIDIA runtime: parse `docker info` JSON output for "nvidia" in runtimes
- The "Override" option allows manually selecting nvidia/amd/intel/none
- On macOS, this screen is informational only (no choices)
- If user chose local Ollama, GPU config is still collected for the compose overlay but the screen
  notes that GPU passthrough only applies to Docker Ollama

```
  AI Model Selection

  Choose which models to download. Models are downloaded after services
  start. You can add more models later.

  LLM Model (for answering questions):

  > [x] qwen2.5:7b          (4.4 GB)  Default, good balance of quality and speed
    [ ] llama3.2:3b          (2.0 GB)  Smaller, faster, less accurate
    [ ] mistral:7b           (4.1 GB)  Strong general-purpose model
    [ ] gemma2:9b            (5.4 GB)  Google's model, good at reasoning
    [ ] phi4:14b             (8.4 GB)  Microsoft's model, needs more RAM

  > [x] nomic-embed-text     (274 MB)  Default, high quality embeddings

**Implementation notes:**

- Uses `huh.NewMultiSelect()` for each category
- At least one LLM and one embedding model must be selected (validation)
- Model sizes are hardcoded in the binary and updated with each release
- The total download size is computed dynamically from selections
- Selected models are joined into a comma-separated string for `OLLAMA_MODELS`
- If local Ollama was chosen, existing models are shown as "already downloaded" with a checkmark

  > sk-...................................................

  database. It never leaves your machine.

**Implementation notes:**

- Keys are temporarily held in wizard state, written to `.env` at the summary step

- Keys are NOT stored in `config.yaml` -- only in `.env` which is gitignored
- The Fernet encryption key is auto-generated and stored in `.env` as `EMBEDINATOR_FERNET_KEY`

## 5. CLI Subcommands Specification

The CLI uses `spf13/cobra` for subcommand routing. The TUI wizard is only activated for `setup`
and `config`; all other subcommands are non-interactive and scriptable.

### 5.1 Command Tree

```
embedinator                     # Default: run setup wizard (if no config.yaml)
                                # or start services (if config.yaml exists)
embedinator setup               # Run the full TUI setup wizard
embedinator start [--dev]       # Start services (docker compose up)
embedinator stop                # Stop services (docker compose down)
embedinator restart [--dev]     # Stop then start
embedinator status              # Show service health table
embedinator logs [SERVICE]      # Stream logs (all or one service)
embedinator config              # Re-run the configuration wizard
embedinator update              # Pull latest images + self-update binary
embedinator doctor              # Diagnose common problems
embedinator version             # Print version and build info
embedinator completion [SHELL]  # Generate shell completions (bash/zsh/fish/powershell)
```

### 5.2 embedinator (no args)

**Behavior:**

```
if config.yaml does not exist:
  run setup wizard (same as `embedinator setup`)
else:
  if services are already running:
    print status table and URLs
  else:
    run `embedinator start` (non-interactive)
```

This makes `embedinator` the single command a user runs after install. First time: wizard. Every
time after: start or show status.

### 5.3 embedinator setup

Runs the full TUI wizard (screens 1-10). If `config.yaml` already exists, the wizard pre-populates
all fields from the existing configuration. The user can modify any setting and re-run the install.

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--non-interactive` | bool | false | Skip TUI, use defaults or existing config.yaml values |
| `--accept-defaults` | bool | false | Same as `--non-interactive` (alias) |

**Non-interactive mode** is for CI/CD or scripted environments. It uses all defaults (or values from
config.yaml) without any prompts. If a required value is missing and has no default, it errors.

### 5.4 embedinator start

Starts all services via Docker Compose.

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--dev` | bool | false | Include dev overlay (hot reload) |
| `--open` | bool | false | Open browser after services are healthy |
| `--timeout` | duration | 2m (5m first run) | Health check timeout |

**Behavior:**

```
1. Load config.yaml (error if not found: "Run `embedinator setup` first")
2. Validate config (error if invalid)
3. Check Docker daemon (error if not running)
4. Check port availability (error with resolution suggestion)
5. Build compose file list (base + gpu overlay + dev overlay + local-ollama overlay)
6. Run: docker compose -f ... up --build -d
7. Poll health endpoints until all healthy or timeout
8. Pull models if not already present
9. Print ready message with URLs
10. If --open: open browser
```

### 5.5 embedinator stop

```
1. Load config.yaml to determine compose file list
2. Run: docker compose -f ... down
3. Print "All services stopped."
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--volumes` | bool | false | Also remove Docker volumes (destructive) |

### 5.6 embedinator restart

```
1. Run stop logic
2. Run start logic
```

Accepts all flags from both `start` and `stop`.

### 5.7 embedinator status

```
1. For each service (qdrant, ollama, backend, frontend):
   a. HTTP health check (same URLs as embedinator.sh)
   b. Docker container status (running/stopped/restarting)
2. Print table:

  Service      Status     Port    Container
  ─────────    ──────     ────    ─────────
  qdrant       healthy    6333    embedinator-qdrant (running)
  ollama       healthy    11434   embedinator-ollama (running)
  backend      healthy    8000    embedinator-backend (running)
  frontend     healthy    3000    embedinator-frontend (running)
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--json` | bool | false | Output as JSON (for scripting) |

### 5.8 embedinator logs

```
1. Run: docker compose -f ... logs -f [SERVICE]
2. Stream output to terminal
```

Passes through to Docker Compose logs directly. Ctrl+C stops streaming.

**Arguments:**

| Arg | Required | Description |
|-----|----------|-------------|
| SERVICE | No | One of: qdrant, ollama, backend, frontend. Omit for all. |

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--tail` | int | 100 | Number of lines to show from the end |
| `--since` | string | "" | Show logs since timestamp (e.g., "1h", "2024-01-01T00:00:00") |

### 5.9 embedinator config

Re-runs only the configuration wizard (screens 3-8 from the setup wizard, skipping welcome and
prerequisites). Pre-populates all fields from existing `config.yaml`. After confirmation, regenerates
`.env` and any overlay files, then offers to restart services if they are running.

### 5.10 embedinator update

```
1. Self-update the embedinator binary:
   a. Check GitHub Releases API for latest version
   b. If newer version available: download, verify checksum, replace binary
   c. If already latest: print "Already up to date (v0.3.0)"

2. Update Docker images:
   a. docker compose pull (all services)
   b. If images changed: offer to restart ("Images updated. Restart services? [Y/n]")
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--self-only` | bool | false | Only update the binary, not Docker images |
| `--images-only` | bool | false | Only update Docker images, not the binary |
| `--yes` | bool | false | Auto-confirm restart |

**Self-update mechanism:**

The binary downloads its replacement to a temp file, verifies the checksum, then uses
`os.Rename` to atomically replace itself. On Windows, this requires a wrapper that:
1. Copies the new binary to `embedinator.new.exe`
2. Spawns a detached process that waits 1 second, renames `.new.exe` to `.exe`
3. Exits the current process

### 5.11 embedinator doctor

Comprehensive diagnostic tool for troubleshooting. Runs all checks and reports findings.

## 6. Configuration Model

### 6.1 config.yaml

The TUI persists its state to `config.yaml` at the repository root. This file captures the user's
wizard choices and allows `embedinator setup` to be re-run without repeating selections.

```yaml
# config.yaml -- Generated by The Embedinator TUI. Do not edit manually.
# Re-run `embedinator config` to modify settings.

version: "1"
created_at: "2026-03-23T14:30:00Z"
updated_at: "2026-03-23T14:30:00Z"
cli_version: "0.3.0"

ollama:
  mode: docker            # docker | local | remote
  remote_url: ""          # Only used when mode=remote
  models:
    llm:
      - qwen2.5:7b
    embedding:
      - nomic-embed-text

gpu:
  profile: nvidia         # nvidia | amd | intel | none
  auto_detected: true     # false if user overrode detection

ports:
  frontend: 3000
  backend: 8000
  qdrant: 6333
  qdrant_grpc: 6334
  ollama: 11434

providers:
  openai: false
  anthropic: false
  openrouter: false

dev_mode: false
```

**Why YAML:**

- Human-readable and editable (even though users should use `embedinator config`)
- Standard Go libraries (`gopkg.in/yaml.v3`)
- Comments are preserved (unlike JSON)
- Aligns with Kubernetes/Docker ecosystem conventions

**The config.yaml is NOT checked into git.** It is added to `.gitignore`.

### 6.2 .env Generation

The TUI generates `.env` from `.env.example` and injects user choices:

| Source | Target .env variable | Notes |
|--------|---------------------|-------|
| config.yaml `ports.frontend` | `EMBEDINATOR_PORT_FRONTEND` | |
| config.yaml `ports.backend` | `EMBEDINATOR_PORT_BACKEND` | |
| config.yaml `ports.qdrant` | `EMBEDINATOR_PORT_QDRANT` | |
| config.yaml `ports.qdrant_grpc` | `EMBEDINATOR_PORT_QDRANT_GRPC` | |
| config.yaml `ports.ollama` | `EMBEDINATOR_PORT_OLLAMA` | |
| config.yaml `gpu.profile` | `EMBEDINATOR_GPU` | |
| config.yaml `ollama.models` | `OLLAMA_MODELS` | Comma-joined |
| Auto-generated | `EMBEDINATOR_FERNET_KEY` | Fernet key via Docker python:3.14-slim |
| Auto-detected | `CORS_ORIGINS` | Computed from frontend port + LAN IP |
| Wizard API key input | Provider keys | Written directly to `.env` |

**Generation logic:**

```
1. If .env exists: read it, preserve all existing values not managed by TUI
2. If .env does not exist: copy .env.example
3. For each TUI-managed variable:
   a. If present in .env: update the value
   b. If not present: append at the end
4. Generate Fernet key if not already set (same Docker-based approach as embedinator.sh)
5. Compute CORS_ORIGINS from frontend port and detected LAN IP
6. Write .env atomically (write to .env.tmp, then rename)
```

### 6.3 Overlay File Generation

When `ollama.mode` is `local` or `remote`, the TUI generates `docker-compose.local-ollama.yml` as
designed in `docs/design-launcher-improvements.md` section 3.6.

This file is NOT checked into git. It is generated by the TUI and referenced only when the compose
file list is built.

### 6.4 Configuration Precedence

From lowest to highest precedence:

```
1. Defaults (hardcoded in the Go binary)
2. config.yaml (persisted wizard choices)
3. .env file (environment variables)
4. System environment variables (export EMBEDINATOR_PORT_FRONTEND=4000)
5. CLI flags (--dev, --timeout, etc.)
```

This matches the precedence model in the existing `embedinator.sh`.

## 9. Relationship to Existing Scripts

### 9.1 Transition Plan

The bash and PowerShell scripts remain in the repository as legacy/fallback entry points. The
transition happens in three phases:

**Phase 1: Coexistence (Initial Release)**

- The Go binary is named `embedinator`
- The scripts remain as `embedinator.sh` and `embedinator.ps1`
- README Quick Start is updated to recommend the binary:
  ```
  # Recommended:
  embedinator

  # Or use the scripts directly:
  ./embedinator.sh          # macOS / Linux
  .\embedinator.ps1         # Windows
  ```
- The binary and scripts can coexist. They both read `.env` and `config.yaml`.
- Users who install the binary will naturally use `embedinator` since it is in their PATH.
- Users who do not install the binary continue using the scripts unchanged.

**Phase 2: Scripts become legacy (After 1-2 minor releases)**

- Scripts are moved to `scripts/legacy/embedinator.sh` and `scripts/legacy/embedinator.ps1`
- A wrapper `embedinator.sh` is placed at the root that prints:
  ```
  [embedinator] The launcher scripts are deprecated.
  [embedinator] Install the TUI binary instead:
  [embedinator]   curl -fsSL https://embedinator.dev/install.sh | bash
  [embedinator] Falling back to legacy script...
  ```
  Then exec's the real script.

**Phase 3: Scripts removed (After 2-3 minor releases)**

- Scripts are removed entirely
- The binary is the only entry point
- `make up` and `make down` continue to work (they use `docker compose` directly)

### 9.2 Feature Parity Matrix

| WSL2 path warning | Yes | Yes | Yes |
| Docker group check (Linux) | Yes | -- | Yes |

## 10. Edge Cases and Failure Modes

### 10.1 Terminal Too Narrow

If terminal width is less than 50 columns, the lipgloss layouts break. Detection:

```go
width, _, err := term.GetSize(int(os.Stdout.Fd()))
if width < 50 {
    // Fall back to a simplified text mode (no borders, no ASCII art)
}
```

The wizard degrades gracefully: ASCII art is replaced with a simple text banner, borders are
removed, and progress bars use simple text (`[=====>    ] 50%`).

### 10.2 No TTY (piped input/output)

When stdin is not a terminal (e.g., `embedinator < /dev/null` or CI), the TUI wizard cannot run.

```go
if !isatty.IsTerminal(os.Stdin.Fd()) && !isatty.IsCygwinTerminal(os.Stdin.Fd()) {
    // Use --non-interactive mode automatically
}
```

All subcommands except `setup` and `config` work fine without a TTY. For `setup` without a TTY,
error with: "Interactive terminal required. Use --non-interactive for headless setup."

### 10.3 Docker Compose File Discovery

The TUI binary must find `docker-compose.yml` and the overlay files. It locates the project root by
searching upward from the current working directory for `docker-compose.yml`:

```go
func findProjectRoot() (string, error) {
    dir, _ := os.Getwd()
    for {
        if _, err := os.Stat(filepath.Join(dir, "docker-compose.yml")); err == nil {
            return dir, nil
        }
        parent := filepath.Dir(dir)
        if parent == dir {
            return "", fmt.Errorf("docker-compose.yml not found (are you inside The Embedinator repo?)")
        }
        dir = parent
    }
}
```

Alternatively, `config.yaml` can store the project root path and the binary can resolve from there.

### 10.4 Binary Installed but Repo Not Cloned

A user runs `embedinator` before cloning the repo. The binary cannot find `docker-compose.yml`.

```
Error: docker-compose.yml not found.

The Embedinator TUI manages a Docker Compose stack that lives in the project repository.

To get started:
  git clone https://github.com/Bruno-Ghiberto/The-Embedinator.git
  cd The-Embedinator
  embedinator
```

### 10.5 Stale config.yaml from a Previous Version

The `version` field in config.yaml allows detecting schema mismatches:

```go
if config.Version != currentConfigVersion {
    // Migrate or re-run wizard
    fmt.Println("Configuration was created by an older version. Re-running setup...")
}
```

Migration logic converts old config schemas to the current version without losing user choices.

### 10.6 Ctrl+C During Installation

If the user hits Ctrl+C during screen 9 (installation progress):

1. The `bubbletea` program catches the interrupt via `tea.KeyMsg`
2. The cleanup handler runs `docker compose down` to stop any partially started services
3. `.env` and `config.yaml` are already written (from screen 8), so the next `embedinator` run
   can resume from start (not re-run the wizard)

### 10.7 Fernet Key Generation Fails

The TUI generates the Fernet key the same way as `embedinator.sh`: via `docker run python:3.14-slim`.
If this fails (no internet, Docker Hub rate limit):

```
Unable to generate Fernet encryption key.

This key is required for encrypting cloud provider API keys.
If you only plan to use local Ollama, you can skip this:

  Press Enter to skip (cloud providers will not work)
  Press r to retry
  Press q to quit
```

If skipped, `EMBEDINATOR_FERNET_KEY` is left empty in `.env` and cloud providers are disabled.

### 10.8 Docker Compose Down Fails (Orphaned Containers)

`embedinator stop` runs `docker compose down`. If containers were started by a different compose
file configuration (e.g., user switched from GPU to non-GPU), orphaned containers may remain.

`embedinator doctor` detects this:

```
[!] Orphaned container found: embedinator-ollama (not in current compose config)
    Fix: docker rm -f embedinator-ollama
    Or: embedinator stop --volumes
```

### 10.9 Self-Update on Windows (Binary in Use)

On Windows, the running binary cannot be replaced. The self-update mechanism:

1. Download new binary to `embedinator.update.exe`
2. Write a small batch file (`_update.bat`) that:
   - Waits 1 second for the current process to exit
   - Renames `embedinator.exe` to `embedinator.old.exe`
   - Renames `embedinator.update.exe` to `embedinator.exe`
   - Deletes `embedinator.old.exe`
   - Deletes itself (`_update.bat`)
3. Spawn the batch file detached
4. Print "Update downloaded. Restart your terminal to use the new version."
5. Exit

### 10.10 Model Pull Timeout

Large models (14B+) can take 30+ minutes on slow connections. The model pull phase does NOT have a
hard timeout -- it streams progress and lets the user Ctrl+C if needed. The progress bar shows
download speed and ETA.

If a model pull is interrupted, Ollama resumes the download on the next attempt (Ollama supports
partial downloads natively).

### 10.11 Running as Root

On Linux, running `embedinator` as root works but is discouraged:

```
WARNING: Running as root is not recommended.
Consider running as a regular user in the docker group:
  sudo usermod -aG docker $USER && newgrp docker
```

The warning is printed but does not block execution.

## 11. Rejected Alternatives

### 11.1 Web-Based Installer (localhost UI)

**Considered:** Serve a web page on localhost during setup, let the user configure in their browser.

**Rejected:** Requires starting a web server before Docker is configured. Creates a chicken-and-egg
problem. Also, opening a browser for a CLI tool feels wrong -- the target audience is comfortable in
the terminal.

### 11.2 Electron/Tauri Desktop App

**Considered:** A native desktop application with a graphical installer.

**Rejected:** Massively increases build complexity, distribution size (100 MB+ vs 10 MB), and
maintenance burden. The terminal is the right environment for a Docker-based tool.

### 11.3 Rust TUI (ratatui)

**Considered:** Rust with `ratatui` and `crossterm` for the TUI.

**Rejected:** The Charm ecosystem in Go produces more visually polished UIs with less code. Rust's
compile times and cross-compilation complexity are disadvantages for a tool that needs rapid
iteration. The project already has Rust in the ingestion worker -- adding Rust to the CLI does not
add value proportional to the added build complexity.

### 11.4 Python TUI (Textual)

**Considered:** Python with `textual` (from the same creator as `rich`).

**Rejected:** Requires Python installed on the user's machine. The entire premise is that Docker is
the ONLY prerequisite. Bundling Python into a standalone executable (PyInstaller, Nuitka) produces
large binaries (50-100 MB) with startup latency and platform-specific quirks.

### 11.5 Bash TUI (dialog/whiptail)

**Considered:** Enhancing `embedinator.sh` with `dialog` or `whiptail` for interactive screens.

**Rejected:** `dialog`/`whiptail` are not installed by default on macOS or Windows. The visual
quality is severely limited (no colors, no progress bars, no animations). Cross-platform behavior
is inconsistent. This approach hits a ceiling very quickly.

### 11.6 Putting the Go Module at the Repo Root

**Considered:** Making `go.mod` live at the repo root instead of `cli/`.

**Rejected:** The repo root contains Python, Rust, and TypeScript projects. A root-level Go module
would confuse tooling (VS Code, dependabot, etc.) and pollute the top-level directory with Go
artifacts. The `cli/` subdirectory cleanly isolates the Go module.

### 11.7 Single Binary That Embeds docker-compose.yml

**Considered:** Embedding `docker-compose.yml` and all overlay files into the Go binary so the user
does not need to clone the repo at all.

**Rejected:** The backend and frontend Dockerfiles reference the source code (build context). Without
the repo, `docker compose build` has nothing to build from. The binary would need to either:
- Clone the repo itself (defeats the purpose of having the binary)
- Use pre-built Docker images from a registry (requires a registry, CI for image publishing, and
  removes the ability to customize)

For now, the binary assumes the repo is cloned. A future version could support a "registry mode"
where pre-built images are pulled directly without a local clone.

### 11.8 Interactive Model Download During Wizard (Before Docker Starts)

**Considered:** Downloading Ollama models during the wizard, before starting Docker services.

**Rejected:** Models are downloaded by the Ollama service running inside Docker (or local Ollama).
The TUI cannot download models without Ollama running. The correct sequence is: start Docker
services first, then pull models.

## Appendix B: Install Script Source (install.sh)

```bash
#!/usr/bin/env bash
# install.sh -- Install the embedinator CLI binary
# Usage: curl -fsSL https://embedinator.dev/install.sh | bash
set -euo pipefail

REPO="Bruno-Ghiberto/The-Embedinator"
BINARY_NAME="embedinator"
INSTALL_DIR="/usr/local/bin"

info()    { echo -e "\033[0;36m[installer]\033[0m $*"; }
success() { echo -e "\033[0;32m[installer]\033[0m $*"; }
error()   { echo -e "\033[0;31m[installer]\033[0m $*" >&2; }
die()     { error "$*"; exit 1; }

# Detect OS and arch
detect_platform() {
  local os arch
  os="$(uname -s | tr '[:upper:]' '[:lower:]')"
  arch="$(uname -m)"

  case "$os" in
    linux)  os="linux" ;;
    darwin) os="darwin" ;;
    *)      die "Unsupported OS: $os" ;;
  esac

  case "$arch" in
    x86_64|amd64)   arch="amd64" ;;
    aarch64|arm64)   arch="arm64" ;;
    *)               die "Unsupported architecture: $arch" ;;
  esac

  echo "${os}_${arch}"
}

# Get latest version from GitHub API
get_latest_version() {
  local url="https://api.github.com/repos/${REPO}/releases?per_page=20"
  curl -fsSL "$url" | grep -oP '"tag_name":\s*"cli/v\K[^"]+' | head -1
}

main() {
  info "Detecting platform..."
  local platform
  platform="$(detect_platform)"
  info "Platform: ${platform}"

  info "Finding latest version..."
  local version
  version="$(get_latest_version)"
  if [[ -z "$version" ]]; then
    die "Could not determine latest version. Check https://github.com/${REPO}/releases"
  fi
  info "Latest version: v${version}"

  local filename="${BINARY_NAME}_v${version}_${platform}.tar.gz"
  local download_url="https://github.com/${REPO}/releases/download/cli/v${version}/${filename}"
  local checksum_url="https://github.com/${REPO}/releases/download/cli/v${version}/checksums.txt"

  info "Downloading ${filename}..."
  local tmpdir
  tmpdir="$(mktemp -d)"
  trap 'rm -rf "$tmpdir"' EXIT

  curl -fsSL -o "${tmpdir}/${filename}" "$download_url"
  curl -fsSL -o "${tmpdir}/checksums.txt" "$checksum_url"

  info "Verifying checksum..."
  (cd "$tmpdir" && grep "$filename" checksums.txt | sha256sum -c --quiet) || \
    die "Checksum verification failed. The download may be corrupted."

  info "Extracting..."
  tar xzf "${tmpdir}/${filename}" -C "$tmpdir"

  # Determine install location
  if [[ -w "$INSTALL_DIR" ]]; then
    info "Installing to ${INSTALL_DIR}/${BINARY_NAME}..."
    mv "${tmpdir}/${BINARY_NAME}" "${INSTALL_DIR}/${BINARY_NAME}"
    chmod +x "${INSTALL_DIR}/${BINARY_NAME}"
  else
    INSTALL_DIR="${HOME}/.local/bin"
    mkdir -p "$INSTALL_DIR"
    info "Installing to ${INSTALL_DIR}/${BINARY_NAME} (no sudo)..."
    mv "${tmpdir}/${BINARY_NAME}" "${INSTALL_DIR}/${BINARY_NAME}"
    chmod +x "${INSTALL_DIR}/${BINARY_NAME}"

    if ! echo "$PATH" | tr ':' '\n' | grep -qx "$INSTALL_DIR"; then
      echo ""
      info "Add ${INSTALL_DIR} to your PATH:"
      info '  echo "export PATH=\"$HOME/.local/bin:\$PATH\"" >> ~/.bashrc'
      info "  source ~/.bashrc"
    fi
  fi

  # Verify
  if "${INSTALL_DIR}/${BINARY_NAME}" version &>/dev/null; then
    success "Installed ${BINARY_NAME} v${version} successfully!"
    echo ""
    info "Next steps:"
    info "  1. Clone The Embedinator:  git clone https://github.com/${REPO}.git"
    info "  2. Enter the directory:    cd The-Embedinator"
    info "  3. Run the installer:      ${BINARY_NAME}"
  else
    die "Installation completed but binary verification failed."
  fi
}

main
```
